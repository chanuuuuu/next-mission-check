# 프로젝트 요구사항 정의서 (PRD) - QR 기반 셀프 체크인 시스템

## 1. 서비스 개요

- **서비스명**: 교회 선교 대원 QR 기반 셀프 체크인 시스템
- **목적**: 오프라인 선교 행사에서 수많은 교회의 참석 대원들이 빠르고 정확하게 도착 상태를 보고하고, 전체 운영진이 실시간으로 현황을 파악할 수 있도록 지원하는 가볍고 강력한 웹 애플리케이션입니다.
- **핵심 가치**:
  - 웹캠 기반의 신속한 QR 스캔 및 모바일 화면 자동 전환을 통한 현장 대기 시간 최소화
  - Edge Runtime 기반 SSE(Server-Sent Events)를 활용한 실시간 대시보드 구현
  - 시스템 장애나 예외 상황 발생 시 수기 입력 및 디스코드 알림을 통한 안정적인 현장 운영 지원

---

## 2. 사용자 및 환경 정의

- **사용자 그룹**:
  - **선교 대원 (모바일 기기 중심)**: 현장에 도착하여 자신의 교회 QR 코드를 생성하고 스캔한 뒤, 체크인 폼을 작성하여 제출합니다.
  - **스태프/현장 관리자 (노트북/PC 중심)**: 안내 데스크에서 노트북 웹캠 화면을 켜두고 대원들의 QR 코드를 스캔해 주거나, 대시보드를 통해 실시간 도착 현황을 모니터링합니다. 문제 발생 시 수동으로 체크인을 처리합니다.
- **운영 환경**: 오프라인 현장 (안정적인 모바일 네트워크 및 와이파이 환경 전제, 참여자 간 상호 신뢰 기반으로 별도의 복잡한 사용자 인증/권한 관리는 생략)

---

## 3. 기능 요구사항 (Functional Requirements)

### [Phase 1] 체크인 QR 코드 생성 페이지 (모바일)

- **교회 선택**: Neon DB에 등록된 N개의 교회 리스트를 화면에 노출하고, 사용자가 자신의 교회를 선택할 수 있도록 합니다.
- **결정적 QR 생성**:
  - 특정 교회를 선택하면 해당 교회를 식별할 수 있는 고유 QR 코드가 생성됩니다.
  - QR 코드의 내용은 `churchId` 값만을 포함하는 결정적(Deterministic) 형태입니다. 동일한 교회는 언제 어느 기기에서 생성하더라도 항상 동일한 QR 코드가 생성됩니다.
  - 스캔 후 모바일 화면 자동 전환은 QR 내부의 데이터가 아닌, 모바일 브라우저가 `churchId` 기반으로 구독하는 SSE 채널을 통해 이루어집니다.
- **QR 코드 다운로드**: 생성된 QR 코드 화면을 이미지 파일(PNG 등)로 기기에 직접 다운로드할 수 있는 버튼을 제공하여, 네트워크 불안정 시 사전 캡처본으로도 스캔이 가능하도록 지원합니다.

### [Phase 2] 체크인 확인 페이지 (노트북 웹캠 스캐너)

- **카메라 스캔**: 노트북에 내장된 웹캠 또는 외장 카메라를 통해 모바일 화면의 QR 코드를 인식합니다.
- **연속 스캔 방지 (디바운싱 및 UI 전환)**:
  - QR 코드가 정상적으로 인식되는 즉시 카메라 스트림 뷰를 화면에서 숨기거나 블러 처리합니다.
  - 화면에 "인식 중입니다... 잠시만 기다려주세요." 등의 로딩 UI를 노출하여 하나의 QR 코드가 중복·연속으로 스캔되는 현상을 원천 차단합니다.
  - 스캔 후 DB 업데이트 API 처리가 완료되면 2초 뒤 카메라 뷰를 다시 활성화하여 다음 대기자를 맞이합니다.
- **예외 처리 및 디스코드 연동**:
  - 유효하지 않은 QR 코드(유효하지 않은 데이터 포맷 또는 DB에 존재하지 않는 교회 정보 등)가 스캔된 경우, 화면 하단에 `"문제가 있습니다. 잠시만 기다려주세요."` 툴팁을 띄웁니다.
  - 이와 동시에, 스캔 실패 로그(실패 시간, 인식된 잘못된 텍스트 등)를 관리자의 **Discord Webhook** 채널로 즉시 전송하여 기술 담당자가 즉각 인지할 수 있도록 합니다.
- **관리자 호출**: 카메라 하단에 항상 "문제가 있어요" 버튼을 고정 노출하여, 시스템 에러 외의 현장 문의가 있을 때 스태프를 호출할 수 있는 시각적 가이드를 제공합니다.

### [Phase 3] 체크인 Form 페이지 (모바일)

- **자동 라우팅**: 노트북 스캐너에서 QR 인식이 완료되면, 해당 `churchId`를 구독하며 대기 중이던 모바일 브라우저가 SSE 이벤트를 수신하여 자동으로 `router.push('/checkin/[churchId]')`를 실행합니다.
- **인사말 출력**: 페이지 진입 시 화면 최상단에 다음 텍스트를 고정 노출합니다.
  - `"안녕하세요 ${교회이름}교회 선교 대원 여러분, 오늘도 너무 고생하셨습니다. 셀프 체크인을 진행하겠습니다."`
- **중복 제출 방지 (Block)**: 관리자가 설정한 현재 활성 Phase(`active_phase`)에서 이미 체크인이 완료된 교회인 경우, 입력 폼 진입을 차단하고 `"이미 체크인이 완료된 교회입니다."` 메시지와 함께 완료 안내 화면만 노출합니다.
- **입력 폼 구성 (추후 질문 항목이 추가될 수 있도록 확장성 고려)**:
  1. **모든 인원 도착 여부 (Checkbox)**: `[] 모든 인원이 도착이 완료되었나요?`
  2. **현재 인원 수 (Number Input)**: 숫자 입력만 허용하며, 유효성 검사는 단순 숫자 포맷 검사만 수행합니다 (현장 변동성을 고려하여 상한선 제한 없음).
  3. **추가 보고 사항 (Textarea)**: `[] 추가적으로 보고가 필요한 사항이 있나요?`
- **완료 버튼**: 폼 최하단에 "완료" 버튼을 배치하며, 클릭 시 데이터가 서버로 전송되고 해당 교회의 실제 도착 여부가 DB에 최종 반영됩니다.

### [Phase 4] 체크인 현황 페이지 (대시보드)

- **화면 레이아웃**: 대형 모니터나 빔프로젝터에 띄우기 적합한 2분할 레이아웃을 채택합니다.
  - **좌측 영역**: 아직 도착하지 않은 (체크인 전) 교회 리스트 노출
  - **우측 영역**: 체크인이 완료된 교회 리스트를 완료된 순서대로 정렬하여 노출
- **노출 포맷**: 도착한 교회의 경우 `${교회명}(${인원수})` 형태로 가독성 있게 표시합니다.
- **현재 Phase 표시**: 대시보드 상단에 현재 활성 Phase 배지를 고정 노출합니다. (예: `2일차 아침 · 2A`) 이 값은 관리자가 설정한 `active_phase` 기준이며, 대시보드의 모든 리스트는 이 Phase 내의 데이터만 표시합니다.
- **실시간 UI 업데이트**: 타임아웃 방식의 긴 폴링 대신, 체크인 완료 DB 트랜잭션이 발생할 때마다 **SSE(Server-Sent Events)**를 통해 대시보드 화면이 실시간으로 즉시 리프레시됩니다. 미도착 리스트에 있던 교회가 도착 리스트로 즉시 이동합니다.
- **Phase 관리**: 총 4일간 진행되는 행사로, **'K일차 아침'**, **'K일차 저녁'** 구조로 **총 8개의 Phase**로 구성됩니다. 관리자가 Phase를 전환하면 대시보드가 해당 Phase 기준으로 즉시 갱신됩니다.

### [Phase 5] 체크인 수동 및 관리 페이지

- **반응형 웹 지원**: 현장 스태프들이 다급하게 움직이며 조작할 수 있도록 모바일 기기와 데스크톱 PC 화면 모두에서 깨짐 없이 원활하게 작동해야 합니다.
- **접근 권한 관리 생략**: 내부 스태프와 대원 간의 신뢰를 바탕으로 운영되므로, 복잡한 로그인이나 로그인 세션 관리는 제외하고 URL 직통 접근 방식을 취합니다.
- **수기 입력 및 수동 제어 기능**:
  - **현재 Phase 전환**: 8개 Phase 코드(`1A` ~ `4P`) 중 현재 활성 Phase를 선택합니다. 변경 즉시 전체 시스템(대시보드 표시 기준, 중복 체크인 판단 기준)이 해당 Phase로 전환됩니다.
  - **교회 추가**: 마스터 데이터에 새로운 교회가 누락되었을 경우 현장에서 즉시 신규 교회를 수동으로 추가할 수 있습니다.
  - **수동 체크인 완료 처리**: QR 코드 손상, 기기 배터리 방전 등 스키마/시스템 오류로 셀프 체크인이 불가능할 경우, 관리자가 목록에서 특정 교회를 선택하고 현장 인원수를 직접 입력하여 즉시 체크인 완료 상태로 강제 변경할 수 있습니다.

## 4. 비기능 요구사항 (Non-Functional Requirements)

- **실시간성 (Real-time)**: 스캔부터 모바일 화면 전환, 그리고 메인 대시보드 반영까지의 지연 시간(Latency)은 2초 이내여야 합니다.
- **안정성 (Vercel Edge Runtime)**: Vercel의 서버리스 함수 타임아웃(15초 제한)을 우회하기 위해 SSE 연결 핸들러는 반드시 Edge Runtime으로 구동하며, 클라이언트의 내장 자동 재연결(Auto-reconnect) 로직을 결합합니다.
- **동시 접속**: 최대 10명의 동시 SSE 접속을 기준으로 설계합니다. 체크인 제출 완료 시 SSE 연결을 즉시 종료하여 리소스를 반환합니다.
- **확장성**: 체크인 폼의 질문 항목 구조는 데이터베이스 스키마 및 JSON 객체 형태로 설계되어 추후 코드를 대대적으로 수정하지 않고도 질문을 추가할 수 있어야 합니다.
