# 기술 스택 정의서 (Tech Stack Specification)

본 프로젝트는 오프라인 현장에서의 실시간성과 Vercel 서버리스 배포 환경의 제약 조건을 모두 만족하기 위해 최신 하이브리드 웹 아키텍처를 채택합니다.

## 1. 프레임워크 및 인프라 (Core Framework & Infrastructure)

- **프레임워크**: `Next.js (App Router)`
  - **선정 이유**: 서버 컴포넌트(RSC)를 통한 빠른 초기 로딩 및 데이터 바인딩(Neon DB 연동)과 클라이언트 컴포넌(`'use client'`)의 풍부한 브라우저 API(웹캠, 카메라스캔, 이미지 다운로드) 제어를 유기적으로 결합할 수 있는 최적의 프레임워크입니다.
- **배포 플랫폼**: `Vercel (Hobby 또는 Pro 요금제)`
  - **특이사항**: 인프라 관리 리소스를 최소화하기 위해 Vercel을 사용하며, 실시간 데이터 스트리밍(SSE) 구현 시 발생하는 서버리스 15초 타임아웃 제약을 극복하기 위해 **Vercel Edge Network(V8 Edge Runtime)** 환경을 100% 활용합니다.
- **데이터베이스**: `Neon DB (Serverless PostgreSQL)`
  - **선정 이유**: Next.js 환경과 완벽하게 호환되는 서버리스 관계형 데이터베이스로, 연결(Connection) 풀링 관리가 용이하며 확장성이 뛰어납니다.

---

## 2. 프론트엔드 아키텍처 (Frontend Architecture)

- **UI 및 스타일링**: `Tailwind CSS` 기반 UI 컴포넌트
  - **개발 방식**: Lovable로 생성된 고품질의 Tailwind CSS 마크업 및 디자인 시스템을 그대로 이식하여 개발 속도를 극대화하고 직관적인 반응형 UI를 제공합니다.
- **서버 상태 관리 (Data Fetching)**: `TanStack Query (React Query) v5`
  - **선정 이유**: 교회 마스터 리스트, Phase 정보 등 서버 데이터의 캐싱, 백그라운드 업데이트, 수동 뮤테이션(`useMutation`) 후 쿼리 무효화(`invalidateQueries`) 처리를 단순화하기 위해 사용합니다. 대시보드 및 세션 체크에 사용됩니다.
- **클라이언트 상태 관리**: `React Context API` 또는 브라우저 내장 상태 (`useState`)
  - **검토 결과**: 핵심 비즈니스 데이터의 90% 이상이 TanStack Query에 의해 서버 상태로 캐싱되므로, Zustand와 같은 별도의 무거운 클라이언트 전역 상태 라이브러리는 필수적이지 않습니다. 카메라 활성화 여부, 툴팁 On/Off 등 순수 UI 상태 관리는 React 내장 기능만으로 충분히 가볍게 구현 가능합니다. (상태 구조가 복잡해질 경우에만 Zustand 도입 고려)

---

## 3. 실시간 통신 및 확장 기능 (Real-time & Integrations)

- **실시간 통신 아키텍처**: `SSE (Server-Sent Events)` 단방향 스트리밍
  - **구현 방식**: Next.js Route Handlers (`app/api/checkin/stream/route.ts`) 상단에 `export const runtime = 'edge';`를 명시하고, 웹 표준 `ReadableStream`을 반환하여 파이프라인을 구축합니다.
  - **동작 원리**:
    1. 노트북 스캐너 스캔 성공시 API 타격 -> 모바일 세션 스트림으로 이벤트 Push -> 모바일 라우팅 실행
    2. 모바일 폼 제출 성공시 DB Insert -> 대시보드 스트림으로 이벤트 Push -> 대시보드 실시간 갱신
- **QR 코드 처리**: `html5-qrcode` 또는 `react-qr-reader` (클라이언트 사이드 라이브러리)
  - 노트북 웹캠 스트림 분석 및 결정적 QR 내부의 세션 토큰 식별을 클라이언트 브라우저 단에서 고속 처리합니다.
- **모니터링 및 알림**: `Discord Webhook API`
  - 유효하지 않은 스캔 데이터가 유입되거나 시스템 예외가 catch 되었을 때, 외부 서드파티 라이브러리 의존성 없이 Next.js 서버단에서 표준 `fetch`를 통해 디스코드 채널로 실시간 경보 메시지를 JSON 스트링 형태로 발송합니다.

## 4. SSR (서버 사이드 렌더링) vs CSR (클라이언트 사이드 렌더링) 하이브리드 전략

| 페이지 / 기능              | 렌더링 방식              | 기술적 이유                                                                                                                                                                                                                  |
| :------------------------- | :----------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **대시보드 (현황 페이지)** | **SSR + 클라이언트 SSE** | 첫 진입 시 Neon DB에서 현재 Phase의 도착/미도착 리스트를 서버에서 미리 그려서(SSR) LCP를 극대화하고, 이후 실시간 변동분만 SSE 클라이언트 리스너가 가볍게 수신하여 화면을 부분 업데이트합니다.                                |
| **QR 코드 생성 페이지**    | **CSR (`'use client'`)** | 사용자가 선택한 교회 ID와 브라우저 단의 세션(LocalStorage/SessionStorage 기반 ID)을 조합하여 동적으로 QR 코드를 드로잉하고 Canvas API를 이용해 이미지 다운로드 기능을 수행해야 하므로 클라이언트 사이드 실행이 필수적입니다. |
| **노트북 스캐너 페이지**   | **CSR (`'use client'`)** | 브라우저 미디어 장치 권한(`navigator.mediaDevices.getUserMedia`)을 획득하여 웹캠 비디오 스트림을 실시간 가공해야 하므로 서버 컴포넌트 환경에서는 동작할 수 없습니다.                                                         |
| **체크인 Form 페이지**     | **CSR 혼합**             | 첫 화면 진입 시 해당 교회 및 Phase의 중복 체크인 여부는 서버 사이드에서 신속히 판단하여 블락 페이지를 리턴할 수 있으며, 실제 데이터 입력 및 인터랙션은 클라이언트 폼 컴포넌트가 담당합니다.                                  |
